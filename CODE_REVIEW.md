## 1. Сериализация кэша через `pickle`

**Что сгенерировал ИИ:** В `app/core/cache.py` класс `RedisCache` сохранял и читал данные через `pickle.dumps` / `pickle.loads`, с запасным вариантом JSON.

**В чём проблема:** `pickle` при десериализации выполняет произвольный код. При компрометации Redis или подмене значения в кэше возможна **удалённая произвольная код-инъекция (RCE)**. Это классическая уязвимость; для кэша API достаточно JSON.

**Как исправил:** Удалил `pickle`, перешёл на `json.dumps` / `json.loads` с `decode_responses=True` и `default=str` для несериализуемых типов.

## 2. Блокирующая команда `KEYS` в Redis

**Что сгенерировал ИИ:** Метод `clear_pattern` использовал `client.keys(pattern)` и затем `delete(*keys)`.

**В чём проблема:** `KEYS` — O(N) по всей базе Redis и **блокирует** сервер на время выполнения. На продакшене с большим числом ключей это приводит к таймаутам и деградации всего кэша.

**Как исправил:** Заменил на итерацию `SCAN` с пакетным `delete`, что не блокирует Redis целиком.

---

## 3. Ошибка `UserRole.admin` в проверке прав

**Что сгенерировал ИИ:** В `app/api/reviews/router.py` при обновлении и удалении отзыва:

```python
current_user.role != UserRole.admin
```

**В чём проблема:** В enum `UserRole` член называется `ADMIN`, а не `admin`. При попытке администратора изменить чужой отзыв возникает **`AttributeError`**, эндпоинт падает с 500 вместо корректной проверки прав.

**Как исправил:** Заменил на `UserRole.ADMIN` в обоих местах.

---

## 4. Сравнение ролей со строками вместо enum

**Что сгенерировал ИИ:** В `app/api/orders/router.py` и `app/api/orders/tracking.py`:

```python
current_user.role not in ["admin", "restaurant_manager"]
```

**В чём проблема:** Поле `role` — это `Enum(UserRole)`, а не строка. Сравнение `UserRole.ADMIN == "admin"` в Python **всегда ложно**. В результате администраторы и менеджеры **не получали расширенных прав**: видели только свои заказы, не могли фильтровать по `user_id`, не имели доступа к чужим заказам и трекингу.

**Как исправил:** Заменил строковые литералы на `(UserRole.ADMIN, UserRole.RESTAURANT_MANAGER)` во всех проверках доступа.

---

## 5. Некорректный подсчёт дубликатов отзывов (SQL NULL)

**Что сгенерировал ИИ:** Проверка «уже оставлял отзыв» фильтровала одновременно `restaurant_id`, `dish_id` и `order_id`, включая `NULL`:

```python
.filter(
    Review.restaurant_id == review_data.restaurant_id,
    Review.dish_id == review_data.dish_id,
)
```

**В чём проблема:** В SQL выражение `NULL = NULL` не истинно. Пользователь мог **многократно оставлять отзывы** на один и тот же ресторан или блюдо — дубликаты не находились.

**Как исправил:** Построение запроса только по ненулевым полям цели (`restaurant_id` или `dish_id`) в `app/api/reviews/router.py` и `app/services/review_service.py`.

## 6. Ошибка `db.func` в админ-статистике

**Что сгенерировал ИИ:** В `app/api/admin/router.py`:

```python
.filter(db.func.date(Order.created_at) == db.func.current_date())
```

**В чём проблема:** У объекта `Session` нет атрибута `func`. При вызове `/api/admin/stats` возникает **`AttributeError`**, эндпоинт полностью неработоспособен.

**Как исправил:** Импортировал `func` из `sqlalchemy` и использовал `func.date` / `func.current_date`. Дополнительно заменил строковые статусы заказов на `OrderStatus` enum.

## 7. Неработающая проверка БД в health-check

**Что сгенерировал ИИ:** В `system_health`:

```python
db.execute("SELECT 1")
```

**В чём проблема:** SQLAlchemy 2.x требует обёртку `text()` для сырого SQL. Без неё запрос падает с исключением, и health-check **всегда** сообщает об ошибке БД, даже при исправном подключении. Timestamp был захардкожен.

**Как исправил:** `db.execute(text("SELECT 1"))`, актуальный `timestamp` через `datetime.now(timezone.utc)`.

## 8. HTTP 404 вместо 401 для удалённого пользователя

**Что сгенерировал ИИ:** В `app/dependencies/auth.py` при валидном JWT, но отсутствующем в БД пользователе, возвращался статус **404 Not Found**.

**В чём проблема:** Нарушение семантики HTTP и OWASP: клиент с действительным токеном должен получать **401 Unauthorized**, а не 404 (утечка информации о существовании ресурса, путаница для клиентов API).

**Как исправил:** Заменил статус на `HTTP_401_UNAUTHORIZED`.

---

## 9. Устаревший `datetime.utcnow()` для JWT

**Что сгенерировал ИИ:** В `app/core/security.py` срок действия токена вычислялся через `datetime.utcnow()`.

**В чём проблема:** `utcnow()` возвращает **наивный** datetime без timezone. В Python 3.12+ метод помечен deprecated; возможны ошибки при сравнении с aware-datetime и при миграции на строгую проверку JWT.

**Как исправил:** `datetime.now(timezone.utc)` для aware UTC-времени.

---

## 10. `create_all` при каждом запуске приложения

**Что сгенерировал ИИ:** В `main.py` безусловный вызов `Base.metadata.create_all(bind=engine)`.

**В чём проблема:** В продакшене это **конфликтует с Alembic**: схема не отражает миграции, возможны расхождения структуры БД, гонки при нескольких воркерах.

**Как исправил:** Вызов обёрнут в `if settings.DEBUG:` — только для локальной разработки.

---

## 11. Неэффективный расчёт среднего рейтинга

**Что сгенерировал ИИ:** В `ReviewService.get_average_rating` загружались **все** оценки в память и усреднялись в Python:

```python
ratings = query.with_entities(Review.rating).all()
total = sum(r[0] for r in ratings)
```

**В чём проблема:** O(n) по числу отзывов, лишняя нагрузка на память и сеть. Для ресторана с тысячами отзывов — заметная деградация.

**Как исправил:** `sql_func.avg(Review.rating)` на стороне СУБД — один агрегирующий запрос.

---

## 12. Устаревший API Pydantic v1 (`dict()`)

**Что сгенерировал ИИ:** В роутерах ресторанов и заказов — `result.dict()`, `restaurant_data.dict()`, `order_data.dict()`.

**В чём проблема:** В Pydantic v2 метод `dict()` deprecated; при обновлении зависимостей код перестанет работать. Для кэша `dict()` без `mode="json"` некорректно сериализует `datetime` и enum.

**Как исправил:** Заменил на `model_dump()` / `model_dump(mode="json")` для кэширования списка ресторанов.
